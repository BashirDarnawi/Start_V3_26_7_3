# Database Migrations (Alembic) — beginner guide

## What this is and why you need it

The app's database has tables (users, sessions, entities, audit_logs,
password_resets). When a future change needs a NEW column or table, simply
deploying new code is not enough — the **existing production database** must
be changed too, carefully, without losing data. That is what migrations do:
each migration is a small, numbered, replayable script that upgrades the
database one step.

Before this existed, the app only ran `create_all` at startup, which can
create *new tables* but silently ignores *changes to existing tables* —
the most likely way to corrupt or crash a deployed app.

## Everyday commands

Run these from the project root (inside Docker use
`docker compose exec albayan ...`):

```
# Show which revision the database is at
alembic current

# Upgrade the database to the latest revision (safe to re-run)
alembic upgrade head

# After changing server/db.py's schema: auto-draft a new migration
alembic revision --autogenerate -m "add phone column to users"
# ... review the generated file in server/migrations/versions/, then:
alembic upgrade head
```

## First run on the production database (one time)

The baseline migration is written so it is **safe on an existing database**
(it only creates missing tables). Just run:

```
alembic upgrade head
```

## Rules

1. **Always back up before migrating production** (pg_dump for Postgres).
2. **Review autogenerated migrations** before running them — autogenerate
   is a draft, not a guarantee.
3. Never edit a migration that has already run in production; write a new
   one instead.
4. Migrations run manually (or from your deploy pipeline) — the app does
   NOT auto-migrate at startup, on purpose: with 2 ECS tasks, concurrent
   auto-migrations could race.
